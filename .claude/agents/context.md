## Constraints & Root Cause Analysis

### **CRITICAL: ICE Server Configuration Missing — ROOT CAUSE OF CROSS-NETWORK FAILURES**
- **File**: `lib/webrtc.ts`, lines 17–19
- **Current configuration**:
  ```typescript
  const ICE_CONFIG: RTCConfiguration = {
    iceServers: [{ urls: "stun:stun.l.google.com:19302" }],
  };
  ```
- **Problem**: STUN server alone cannot traverse symmetric NAT (present on many enterprise networks and carrier-grade NAT)
- **Why same-WiFi works**: Both peers share same NAT gateway; STUN finds local IP, hairpinning succeeds
- **Why cross-WiFi fails**: Different NAT contexts; STUN finds external IP for each peer, but symmetric NAT blocks inbound—no reflexive candidate is usable; ICE gathers candidates but all fail
- **User-visible symptom**: Peers see "network error" or "user decline" (ICE connection fails before datachannel opens)
- **Solution required**: Add TURN server with credentials. Config must be:
  ```typescript
  const ICE_CONFIG: RTCConfiguration = {
    iceServers: [
      { urls: "stun:stun.l.google.com:19302" },
      {
        urls: ["turn:your-turn-server.com:3478"],
        username: "user",
        credential: "pass",
      },
    ],
  };
  ```
- **TURN server options**: Coturn (self-hosted), Xirsys (managed), AWS Kinesis Video Streams, or other commercial TURN provider
- **Firewall requirement**: Both client and server must support UDP 3478–3479 (STUN/TURN standard ports)
