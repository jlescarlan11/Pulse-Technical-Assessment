## Constraints

### Product / privacy rules (from docs/requirements.md, README, schema comments)
- **No accounts, no PII, stateless by design.** Nothing persists between sessions. No message/call/connection history.
- **Chat and video are P2P only** ‚Äî they must never reach or be stored by the server.
- **Location must be privacy-offset 1‚Äì3 km**, randomized fresh each session; raw coords are never stored (enforced server-side in `app/api/join/route.ts` via `applyPrivacyOffset`).
- **One active connection at a time** per user (enforced via `busy` flag in `app/api/signal/route.ts`).
- **Deploy to Vercel only**, single Next.js project, no extra third-party platforms (Postgres, Mapbox, Cloudflare TURN are the allowed externals).
- `AGENTS.md`: this Next.js version differs from training data ‚Äî read in-repo docs before writing Next.js code.

## Phase 3 focus: API surface & security-relevant notes

### Auth / identity model
- **No authentication and no tokens.** Identity is a **client-generated session id** (a UUID, per schema comment "client-generated session uuid"). The client supplies its own `id` on `join` and passes it as `id`/`fromId`/`toId` on every other call.
- **Trust boundary**: the server trusts whatever id the client sends. There is **no proof a caller owns the id** it claims. Implications to flag (not fix):
  - Any client can poll/drain another user's mailbox by passing `?id=<victim>` to `/api/poll` (signals are deleted on read ‚Üí message theft / denial of delivery).
  - Any client can spoof `fromId` on `/api/signal` to impersonate a peer, or set another user's `busy` state, or inject `end`/`decline` to disrupt a connection.
  - Any client can call `/api/leave` with another user's id to evict them.
  - `id` is fully attacker-chosen on `join` (any 8‚Äì64 char string), so ids are guessable/enumerable if not random.

### Routes (inputs ‚Üí DB effects)
1. **POST `/api/join`** ‚Äî body `{ id, lat, lng }` (raw coords). Validates `id` (string, len 8‚Äì64) and lat/lng ranges (`isValidLatLng`). Applies 1‚Äì3 km privacy offset, then `prisma.presence.upsert` on `id`. Stores only offset coords. Returns `{ ok: true }`.
2. **GET `/api/poll?id=`** ‚Äî query `id` (presence-only check, no format validation). Does four DB ops: (a) `presence.updateMany` heartbeat for caller; (b) reap `presence.deleteMany` where `lastSeen < now-STALE_MS` and `signal.deleteMany` where `createdAt < now-SIGNAL_TTL_MS`; (c) `presence.findMany` of other online peers (id/lat/lng/busy); (d) `signal.findMany` where `toId = id` then `deleteMany` those ids (read-and-delete mailbox drain). Returns `PollResponse { peers, signals }`.
3. **POST `/api/signal`** ‚Äî body `{ fromId, toId, type, payload? }`. Validates `fromId`/`toId` are strings (no length/format), `type` in enum (request|accept|decline|offer|answer|ice|end), `payload` optional string ‚â§ 64 KB. For `request`: looks up target presence; if missing or `busy`, auto-declines (creates a `decline` signal back) instead of delivering. For `accept`: marks both peers `busy = true`. For `decline`/`end`: marks both `busy = false`. Always `signal.create` the message. Returns `{ ok: true }` (or `{ ok, autoDeclined }`).
4. **POST `/api/leave`** ‚Äî body `{ id }` (parsed defensively as text/JSON for `sendBeacon`). Validates `id` is non-empty string. `signal.deleteMany` where `toId=id OR fromId=id`, then `presence.deleteMany` where `id`. Returns `{ ok: true }`.
5. **GET `/api/turn-credentials`** ‚Äî no input. Reads `CLOUDFLARE_TURN_TOKEN_ID` + `CLOUDFLARE_TURN_API_TOKEN`; POSTs to Cloudflare Realtime TURN to mint ICE creds (ttl 86400s), returns `{ urls, username, credential }` with `Cache-Control: private, max-age=300`. No DB. Fails closed (500) if unconfigured or upstream invalid. **No auth/rate-limit** ‚Äî any caller can mint TURN credentials (cost/abuse vector to flag).
6. **`lib/api.ts`** ‚Äî client wrappers: `join` (POST JSON), `poll` (GET, no-store, throws on non-OK), `sendSignal` (POST JSON), `leave` (navigator.sendBeacon, fallback keepalive fetch). No retry/backoff; no auth headers.

### Data layer / TTL & cleanup
- **Presence** (`id` PK = session uuid, `lat`, `lng`, `busy`, `lastSeen`, index on `lastSeen`): only offset coords stored. Heartbeated on each `poll`.
- **Signal** (`id` PK uuid, `toId`, `fromId`, `type`, `payload?` JSON string, `createdAt`, index on `toId`): mailbox for connection + WebRTC signaling only.
- **TTL/cleanup is lazy, piggybacked on `/api/poll`** (no cron): presence reaped after `STALE_MS = 15s` without heartbeat; orphan signals reaped after `SIGNAL_TTL_MS = 60s`; delivered signals deleted on read; `leave` does explicit cleanup. Client poll cadence `POLL_INTERVAL_MS = 1.5s`. Cleanup only runs when *someone* polls ‚Äî if no users are online, stale rows persist until the next poll.

### Open security questions to carry into Phase 3
- No caller-owns-id verification ‚Üí spoofing/eviction/mailbox-theft across all id-bearing routes.
- No rate limiting anywhere (join flooding, signal spam, unbounded TURN credential minting).
- No security headers / CSP. `payload` is stored and relayed as opaque string (size-capped) ‚Äî only a risk if any consumer renders it as HTML (WebRTC SDP/ICE only, so low, but worth confirming).
- Hard-coded Mapbox fallback token in `WorldMap.tsx` and `CLOUDFLARE_TURN_*` missing from `.env.example`.
